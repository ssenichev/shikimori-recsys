# Рекомендательная система аниме по данным Shikimori

## Идея

У нас есть данные с платформы Shikimori: каталог аниме с описаниями и оценки пользователей (1-10). Задача — по истории просмотров пользователя предсказать, какое аниме ему понравится дальше.

Система работает в два шага:
1. **Быстрый поиск** (Two-Tower): из всего каталога (~10K аниме) отбираем 100 кандидатов за ~10мс
2. **Точное ранжирование** (Cross-Encoder Reranker): среди 100 кандидатов выбираем топ-10 за ~200мс

Обучение идёт в три стадии, каждая решает свою задачу.

---

## Данные

### Что у нас есть

- **anime.csv** — каталог аниме: название, описание, жанры, студия, рейтинг
- **users_rates.csv** — взаимодействия: кто, что и как оценил, сколько серий посмотрел, когда
- **genres.csv** — справочник жанров

### Как мы обрабатываем данные

**Текстовое представление аниме**: Для каждого аниме склеиваем в одну строку: название + русское название + жанры + **полное описание** из каталога Shikimori (очищенное от BBCode-тегов). Это то, что будет "читать" языковая модель. Описание используется целиком, единственное ограничение — токенизатор обрезает до 512 токенов (~300-400 слов), чего хватает для подавляющего большинства аниме.

```
Fullmetal Alchemist: Brotherhood (Стальной алхимик: Братство)
Genres: Action, Adventure, Drama, Fantasy.
Жизнь братьев Элриков текла спокойно и безмятежно, пока
мальчики не решились на запретное — воскресить умершую мать
с помощью алхимии. Однако попытка человеческого преобразования
привела к ужасным последствиям...
```

**Два типа взаимодействий**:

| Тип | Что это | Как используем |
|---|---|---|
| **Явные** (explicit) | Пользователь поставил оценку 1-10 | Основной сигнал. Оценка >= 7 = "понравилось", <= 4 = "не понравилось" |
| **Неявные** (implicit) | Посмотрел, но не оценил | Контекст. Если досмотрел >= 80% серий — считаем, что понравилось (ставим оценку 7) |

**Почему мы "повышаем" неявные взаимодействия**: Если человек досмотрел аниме почти до конца и не бросил — скорее всего оно ему понравилось. Это сильно увеличивает количество положительных примеров для обучения.

### Разбиение на train/val/test

Используем **временной сплит**: для каждого пользователя берём его историю в хронологическом порядке и прячем последние два аниме с высокой оценкой (>= 7):

```
История пользователя:
[аниме_A: 5/10] [аниме_B: 8/10] [аниме_C: 3/10] [аниме_D: 9/10] [аниме_E: 7/10]
 \___________________ train _________________/   \___ val  ___/ \___ test  __/
```

Так мы проверяем: может ли модель предсказать будущее по прошлому. Это честная оценка, без утечки данных из будущего.

## Стадия 1: Обучение текстового энкодера (Contrastive Fine-tuning)

### Задача

Научить языковую модель понимать, что "похожие по вкусу" аниме должны иметь близкие эмбеддинги (векторные представления), а непохожие — далёкие.

### Модель

Берём предобученный **multilingual-e5** (мультиязычная модель для получения текстовых эмбеддингов, основана на XLM-RoBERTa). Дообучаем через **LoRA** — это техника, которая замораживает основные веса модели (~300M параметров) и добавляет маленькие обучаемые адаптеры (~0.5M параметров) в слои внимания. Так мы можем дообучить большую модель на маленьком датасете не переобучившись.

Архитектура:
```
Текст аниме → XLM-RoBERTa (замороженная + LoRA адаптеры)
            → Mean pooling по токенам
            → Linear(1024, 256) + LayerNorm
            → L2-нормализация
            → 256-мерный вектор (эмбеддинг)
```

### Данные

Для каждого пользователя строим тройки (anchor, positive, negative):
- **Anchor**: аниме со средней оценкой (5-7) — "отправная точка"
- **Positive**: аниме с высокой оценкой (>= 7) — "такое же хорошее"
- **Negative**: аниме с низкой оценкой (<= 4) — "плохое"

Важно: неоценённые аниме (implicit) не используются как негативные примеры — "не смотрел" не значит "не понравилось".

### Лосс

**Triplet Margin Loss** с адаптивным отступом:

```
d_ap = 1 - cosine(anchor, positive)   # расстояние до хорошего
d_an = 1 - cosine(anchor, negative)   # расстояние до плохого
margin = 0.3 * (score_pos - score_neg) / 9   # чем больше разница оценок, тем больше отступ
loss = max(0, d_ap - d_an + margin)
```

Модель учится: расстояние до "хорошего" аниме должно быть меньше расстояния до "плохого" минимум на margin. Пара (9/10 vs 2/10) получает больший margin, чем (7/10 vs 5/10) — так модель учитывает силу предпочтения.

### Почему это работает

После обучения модель создаёт "пространство вкусов": аниме, которые нравятся одним и тем же пользователям, оказываются рядом. При этом модель понимает содержание (жанры, сюжет, описание), а не просто запоминает ID.

## Стадия 2: Two-Tower (Двухбашенная модель)

### Задача

Научиться по истории просмотров пользователя предсказывать, какое аниме ему понравится следующим. Нужно уметь быстро искать по всему каталогу.

### Архитектура

Две "башни" создают эмбеддинги в одном 256-мерном пространстве:

**ItemTower** (замороженная):
- Текстовый энкодер из Стадии 1 — уже обучен, не меняется
- Один раз кодируем все аниме в каталоге → матрица эмбеддингов `[N_items, 256]`
- Опционально: + CF-эмбеддинг (обучаемый вектор на каждое аниме, пропорция регулируется гейтом ~10%)

**UserTower** (обучаемая):
```
Для каждого аниме из истории пользователя:
    эмбеддинг_аниме (из кэша) + проекция_оценки(score) => контекстный вектор

TransformerEncoder (2 слоя, 4 головы внимания)
    => Attention между аниме из истории
    => Mean pooling по валидным позициям
    => Linear + LayerNorm + L2-нормализация
    => 256-мерный вектор пользователя
```

Трансформер внутри UserTower "смотрит" на всю историю пользователя и решает, какие аниме из неё важнее для предсказания. Оценки (score) проецируются в пространство эмбеддингов и прибавляются — так модель знает, что пользователь оценил на 10, а что на 3.

### Почему ItemTower заморожена

Если бы ItemTower обучалась вместе с UserTower, возникла бы проблема **стухшего кэша**: контекстные эмбеддинги (из кэша, посчитанного в начале эпохи) и целевые эмбеддинги (из свежей модели) оказались бы в разных пространствах. Модель не может учиться, если левая и правая части сравнения измеряются в разных единицах.

Замораживание решает это: кэш строится один раз и всегда актуален.

### Данные

Для каждого "понравившегося" аниме (score >= 7) строим пример:
- **Контекст**: все аниме, которые пользователь смотрел **до** этого (строго хронологически), до 50 штук
- **Цель**: то самое аниме, которое нужно найти

```
Контекст: [Naruto (8/10), Death Note (9/10), SAO (5/10), ...]
Цель:     Steins;Gate
Вопрос:   Может ли модель "вытащить" Steins;Gate из 10K аниме, зная контекст?
```

### Лосс

**In-batch Sampled Softmax + Hard Negatives**:

1. В батче B пользователей, каждый с целевым аниме
2. Считаем `user_embs @ target_embs.T / temperature` → матрица `[B, B]`
3. Правильный ответ — диагональ (пользователь i → аниме i)
4. Все остальные аниме в батче — **in-batch negatives** (чужие цели)
5. Дополнительно: ищем **hard negatives** — аниме из каталога, которые модель ошибочно считает похожими на пользователя (top-128 по скору, исключая правильный ответ)
6. Склеиваем: `[B, B+128]` логитов → cross-entropy

```
temperature = 0.07 — "заостряет" распределение, усиливая различия между кандидатами
```

### Почему hard negatives важны

Без них модель видит ~63 случайных негативных примера на батч (чужие цели). Это слишком легко — случайное аниме обычно сильно отличается от предпочтений пользователя. Hard negatives заставляют модель различать "похожее, но не то" от "то, что нужно" — это гораздо сложнее и полезнее.

## Стадия 3: Cross-Encoder Reranker

### Задача

Two-Tower быстро отбирает 100 кандидатов через dot-product, но это грубое сравнение — два независимых вектора не могут уловить тонкие взаимосвязи ("любит меху, но только с психологическими темами"). Реранкер смотрит на пару (пользователь, аниме) целиком и даёт точную оценку.

### Модель

**BAAI/bge-reranker-v2-m3** — предобученная модель для ранжирования текстовых пар. Уже умеет оценивать релевантность, мы дообучаем на наших данных.

Вход:
```
[CLS] Enjoyed: Attack on Titan (10), Steins;Gate (9) | Disliked: School Days (2) [SEP] <описание аниме-кандидата> [SEP]
```

Профиль пользователя — это текстовое описание вкусов (топ-8 любимых, топ-3 нелюбимых, 5 просто просмотренных), а не сырая история. Так вмещаемся в 512 токенов и даём модели понятный контекст.

Выход: `sigmoid(logits)` → число от 0 до 1 (оценка релевантности).

### Данные

Каждое явное взаимодействие пользователя → пара (профиль, аниме) с целевой оценкой `score_norm ∈ [0, 1]`. Плюс 3 случайных аниме, которые пользователь не смотрел, с оценкой 0.

### Лосс

**MSE (Mean Squared Error)** — регрессия:

```
loss = (predicted_score - true_score_norm)²
```

Модель учится предсказывать, насколько аниме понравится пользователю по шкале 0-1.

### Почему именно cross-encoder

В Two-Tower модели пользователь и аниме кодируются **независимо** — они "не видят" друг друга до финального dot-product. Cross-encoder пропускает оба текста через один трансформер — attention слои могут находить сложные взаимосвязи между конкретными предпочтениями пользователя и конкретными свойствами аниме. Это медленнее (нельзя использовать предвычисленный кэш), но точнее.

## Почему это будет работать

### Мультиязычность
Shikimori — русскоязычная платформа, но описания аниме часто на английском/японском. multilingual-e5 обучен на 100+ языках и понимает все три.

### Контентная + коллаборативная фильтрация
- **Контентная** (текстовые эмбеддинги): модель понимает, что "Steins;Gate" и "Re:Zero" похожи по описанию (sci-fi, путешествия во времени)
- **Коллаборативная** (CF эмбеддинги): модель знает, что пользователи, которым нравится X, часто любят и Y — даже если описания X и Y не похожи
- Оба сигнала объединяются через гейт: `fused = text_emb + sigmoid(gate) * cf_emb`

### Трёхстадийное обучение
- **Стадия 1** (триплетный лосс): создаёт хорошее текстовое пространство, где "похожие по вкусу" аниме рядом
- **Стадия 2** (sampled softmax): учит строить вектор пользователя, который ищет правильное аниме среди тысяч кандидатов
- **Стадия 3** (MSE-регрессия): точно ранжирует финальных кандидатов, видя и пользователя и аниме одновременно

Каждая стадия использует результат предыдущей: Стадия 2 берёт энкодер из Стадии 1, Стадия 3 берёт кандидатов из Стадии 2.

### Hard negatives + заморозка
Без hard negatives модель легко отличает "Наруто" от "документальный фильм про рыб", но не может отличить "Mob Psycho 100" от "One Punch Man" (оба — аниме про супергероев от одного автора). Hard negatives именно это и тренируют.

Заморозка ItemTower гарантирует, что пространство эмбеддингов стабильно: кэш всегда актуален, лосс уменьшается монотонно.

### Временной сплит
Мы не перемешиваем данные случайно — прячем последние по времени взаимодействия. Это имитирует реальную задачу: предсказать, что понравится пользователю в будущем, зная только прошлое.

## Метрики

| Метрика | Что измеряет |
|---|---|
| **HR@k** (Hit Rate) | Доля пользователей, для которых правильное аниме попало в топ-k |
| **NDCG@k** | То же, но чем выше в списке — тем лучше (позиция 1 ценнее позиции 10) |
| **MRR** | Среднее 1/ранг — насколько высоко в среднем оказывается правильный ответ |
| **Retrieval Recall@100** | Доля пользователей, для которых Two-Tower нашёл правильное аниме в топ-100 (потолок для реранкера) |

При оценке **маскируем уже просмотренные** аниме — не считаем за успех, если модель рекомендует то, что пользователь уже видел.
